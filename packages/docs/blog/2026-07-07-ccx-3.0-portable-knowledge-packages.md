You spend weeks building a knowledge graph. Upload the documents, confirm the domain, let extraction run, watch entities and relationships take shape, clean up the graph until it's actually useful. Then you want to move it -- to a new machine, to a colleague's instance, to a backup you keep off-site. Right now, for most tools in this space, that knowledge is welded to wherever it was built.

<!-- truncate -->

Chaos Cypher's answer is CCX: a single `.ccx` file that bundles your templates, entities, relationships, citations, and source text into one portable package. Export it from one instance, load it into another, and the graph reconstructs -- provenance intact. This post covers what's actually in a CCX package, how to export and load one today, and what's next for sharing packages with the wider community.

## What's in a Package

A `.ccx` file is a compressed archive. Depending on what you choose to include, it can carry:

| Component | What it holds |
|---|---|
| **Templates** | Your node and edge type definitions -- the schema of your graph |
| **Knowledge** | Entities and relationships, with their properties |
| **Sources** | Document metadata, chunks, citations, and tags -- the provenance behind every entity |
| **Workflows** | Automation definitions and triggers |

Everything is optional and toggled independently -- export just the schema to share a domain template, or the full graph including source text for a complete handoff. Embeddings are excluded by default (they're only useful if the receiving instance uses the same embedding model) but can be included for same-model migrations.

**In plain English:** a CCX package is your knowledge graph, minus the app -- open it on another Chaos Cypher instance and it's whole again.

## Exporting and Loading a Package

The most common path is the Settings page. Open **Settings → General → Import & Export**, pick which components to include, and click **Export** -- your browser downloads the `.ccx` file.

<!-- screenshot: Settings > Import & Export page with component checkboxes and the Export button -->

Loading a package works the same way in reverse: **Import**, select the file, choose whether to merge with or replace existing data, and Chaos Cypher rebuilds the graph -- templates first, then entities and relationships, then sources.

If you only want one document's worth of knowledge -- not the whole graph -- open that source's action menu on the Sources page and choose **Export Source** instead. That produces a `.ccx` scoped to just the entities, relationships, and citations traceable back to that one document.

**In plain English:** export is a button, import is a button, and what comes out the other side is the same graph you started with.

## What's Actually Inside the File (for the curious)

Under the hood, a CCX package is JSON-LD, not a proprietary binary blob. Unzip one and you'll find a `manifest.json` describing what's inside and how it was generated, `knowledge.jsonld` and `templates.jsonld` for the graph itself, `sources.jsonl` for document and citation data, and a human-readable `README.txt`. Chaos Cypher-specific concerns -- editor layout, workflow definitions, UI-only metadata -- live in clearly namespaced `chaoscypher.*` graphs inside the package, so a generic JSON-LD or RDF tool can still read the parts that follow the open vocabulary and safely ignore the parts that don't.

That separation is deliberate. We want `.ccx` to be an **open, documented format** -- not a lock-in mechanism. There's a permissively licensed (Apache-2.0) standalone reader on PyPI, `ccx-format`, that reads a package without needing Chaos Cypher installed at all:

```bash
pip install ccx-format
python -c "import ccx; pkg = ccx.load('my-knowledge.ccx'); print(pkg.manifest)"
```

We're calling it an open format, deliberately not a "standard" yet -- that word is earned once a second, independent tool reads or writes CCX packages, not before.

**In plain English:** the format is documented and the reader is open source, so your knowledge graph is never trapped behind one piece of software.

## For CLI and Automation Users

Everything above has a scriptable equivalent. The CLI exports and loads packages directly:

```bash
# Export the current database's templates, knowledge, lenses, and workflows
chaoscypher graph package export --output my-knowledge.ccx

# Load it into another instance (or another local database)
chaoscypher graph package load my-knowledge.ccx --database research
```

Toggle components the same way as the UI: `--no-workflows`, `--no-knowledge`, `--lens-id <id>` for a single lens, and so on (`chaoscypher graph package export --help` for the full list). One gap worth flagging honestly: the CLI export path doesn't yet wire up source data, so a `.ccx` built with `graph package export` is knowledge-only -- for a package that includes sources and citations, export from the Settings page or the API for now.

The REST API is async, which matters if you're scripting exports of a large graph:

```bash
# Kick off a full export
curl -X POST "http://localhost:8080/api/v1/exports?include_sources=true"
# -> {"task_id": "..."}

# Poll until it's done, then download
curl "http://localhost:8080/api/v1/queue/tasks/{task_id}/result" -o my-knowledge.ccx
```

Importing follows the same poll-and-fetch pattern against `POST /api/v1/exports/import`. Full parameter reference is in the [Import & Export docs](/docs/user-guide/import-export).

**In plain English:** if you'd rather script it than click it, every export/import path is one `curl` or one CLI command away.

## What's Next: Lexicon Hub

Exporting and loading between your own instances is live today. The next step is Lexicon Hub -- a community registry where you'll be able to publish a package publicly, discover ones other people have built, and pull them straight into your own graph as a starting point. Think of it as the "share your research graph like a GitHub repo" moment: a medical-ontology template someone else spent a week refining, or a fully extracted knowledge base on a topic you're just getting started with.

The Hub isn't generally available yet -- the client-side plumbing (CLI commands, API contract) is built and documented, but the hosted service itself is still coming online. We wanted the CCX format itself to be solid, dogfooded, and genuinely portable before layering a community registry on top of it. Watch this space, or read the [Lexicon Hub docs](/docs/lexicon-hub) for the shape of what's coming.

Until then: export a package, hand it to a teammate, load it into your other machine. That part works today, and it's the part that matters most.
